# ADO Test Reporter: User Guide for Exploratory Testers

Welcome to the **ADO Test Reporter**! This tool is designed to make your life easier by automatically syncing your test results, feature files, and Excel test cases directly into Azure DevOps (ADO) Test Plans. This means less manual data entry and more time for actual exploratory testing.

This guide will walk you through the core features step-by-step.

---

## Prerequisites

Before you begin, ensure you have the following setup:

1.  **Node.js**: Ensure Node.js (version 18 or higher) is installed on your machine. You can check this by opening a terminal and running `node -v`.
2.  **Azure DevOps Access**: You need access to an Azure DevOps Organization and Project.
3.  **Personal Access Token (PAT)**: You need a PAT with **Read/Write** access to **Work Items** and **Test Management**.
4.  **Configuration File (`.env`)**:
    *   In the root of the project folder, create a file named `.env` (you can copy `.env.example` if it exists).
    *   Add your connection details:
        ```ini
        ADO_ORG_URL=https://dev.azure.com/your-org
        ADO_PROJECT=YourProject
        ADO_TOKEN=your_pat_token
        ```
5.  **Install Dependencies**: Open your terminal in the project folder and run:
    ```bash
    npm install
    ```

## Feature 1: Importing Test Cases from Excel

If you have existing test cases in an Excel spreadsheet, you can quickly import them into ADO.

### Instructions

1.  **Prepare your Excel File**: Ensure your Excel file has clear column headers (e.g., Title, Description, Steps).
2.  **Create a Mapping File**: Create a JSON file (e.g., `mapping.json`) that tells the tool which Excel columns map to which ADO fields.
    *   *Example `mapping.json`:*
        ```json
        {
          "Title": "System.Title",
          "Description": "System.Description",
          "Test Step": "Microsoft.VSTS.TCM.Steps"
        }
        ```
3.  **Open your Terminal** in the project folder.
4.  **Run the Import Command**: Execute the following command, replacing the placeholders with your actual file paths:
    ```bash
    npm run import-excel -- --file "path/to/your/testcases.xlsx" --mapping "path/to/mapping.json"
    ```
    *   *Optional*: You can specify a Test Plan and Suite by adding `--plan-name "My Plan" --suite-name "My Suite"`.
5.  **Verify in ADO**: Go to Azure DevOps -> Test Plans. **Navigate** to the Test Plan and Suite you specified (or the default ones) to see your imported test cases.

> **[Screenshot: Main ADO Test Plans view highlighting the newly imported test cases in the list.]**

### 💡 Pro-Tips
*   **Data Cleaning**: Make sure your Excel data is clean. The tool automatically sanitizes inputs to prevent issues, but clean data imports faster.
*   **HTML Fields**: If you have columns with rich text (like bold or lists), ensure they are mapped to fields that support HTML in ADO (like `System.Description`).

## Feature 2: Syncing Gherkin Feature Files

If your team uses BDD (Behavior-Driven Development) and writes `.feature` files, you can sync these directly to ADO Test Cases.

### Instructions

1.  **Tag your Feature Files**: Ensure the Scenarios in your `.feature` files have a tag corresponding to an ADO Test Case ID (e.g., `@TC123`).
    *   *Example:*
        ```gherkin
        @TC123
        Scenario: User logs in successfully
          Given the user is on the login page
          When they enter valid credentials
          Then they are redirected to the dashboard
        ```
2.  **Open your Terminal** in the project folder.
3.  **Run the Sync Command**:
    ```bash
    npm run sync-features
    ```
    *(Note: This scans the `features/**/*.feature` directory by default).*
4.  **Verify in ADO**: Open Test Case `123` in Azure DevOps. You should see the `Given/When` steps mapped to the "Action" and the `Then` steps mapped to the "Expected Result".

> **[Screenshot: ADO Test Case view showing the Steps grid populated with Given/When/Then steps.]**

### 💡 Pro-Tips
*   **Single Source of Truth**: Treat your feature files as the master copy. Changes made in the feature file will overwrite the ADO Test Case steps on the next sync.
*   **Shared Steps**: Backgrounds in feature files are automatically prepended to every scenario, saving you from duplicating setup steps.

## Feature 3: Publishing Test Results (JUnit)

After running automated tests (or manually formatting results into a JUnit XML file), you can publish the results to ADO. Failed tests can even auto-create defect tasks!

### Instructions

1.  **Locate your JUnit XML**: Find the `.xml` file generated by your test runner (e.g., `results.xml`).
    *   *Crucial*: The `<testcase>` names in the XML *must* include the ADO Test Case ID (e.g., `name="Login Test_TC123"`).
2.  **Open your Terminal** in the project folder.
3.  **Run the Publish Command**:
    ```bash
    npm start -- --junit-file "path/to/results.xml" --plan-name "Exploratory Run 1" --suite-name "Build 45"
    ```
4.  **Check ADO Runs**: Go to Azure DevOps -> Test Plans -> Runs. **Click** on the recent run to view the pass/fail status.
5.  **Check Backlog (for failures)**: If any tests failed, go to your ADO Boards -> Work Items. You should see new "Task" (or "Bug") items created with the failure details.

> **[Screenshot: ADO Work Item view showing an auto-generated Task for a failed test, highlighting the failure message in the description.]**

### 💡 Pro-Tips
*   **Auto-Generate Plans**: Use `--plan-name auto-generate` to create a fresh Test Plan for every run, keeping your historical data pristine.
*   **Smart Defect Logging**: The tool prevents duplicate bugs! If a test fails again, it will just add a comment to the existing open Task instead of creating a new one.

## Troubleshooting Common Errors

*   **Error: `Access Denied` or `401 Unauthorized`**
    *   *Fix*: Check your `.env` file. Ensure `ADO_TOKEN` is correct, hasn't expired, and has the necessary permissions (Read/Write for Work Items and Test Management).
*   **Error: `Project does not exist` or `404 Not Found`**
    *   *Fix*: Verify the `ADO_ORG_URL` and `ADO_PROJECT` in your `.env` file exactly match your Azure DevOps URLs.
*   **Tests are not linking to ADO IDs**
    *   *Fix*: Ensure your test names in the JUnit XML or tags in your Feature files strictly follow the format (e.g., ending with `_TC123` or tagged `@TC123`).
*   **Excel Import Fails due to "Invalid Mapping"**
    *   *Fix*: Double-check your `mapping.json`. Ensure the ADO field names (e.g., `System.Title`) are spelled correctly.
